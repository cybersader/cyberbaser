This page is the map. Every design decision in cyberbaser, and every harder document in [Design](/cyberbaser/design/) and [Research](/cyberbaser/research/), reduces to the eight primitives below. Read them once and the rest of the project stops looking like a pile of features and starts looking like a few ideas applied over and over.

Two of them carry more weight than the rest. **Translation Layer** and **Round-Trip Editability** are the load-bearing walls: get them right and the contribution paths, stable URLs, and single source of truth all compose cleanly; get them wrong and the rest is aspiration. Each entry is the same shape (what it is, why it matters, how cyberbaser uses it) followed by a small picture built from real interface, plus a link into the page that goes deeper.

## Vault

<div class="cb-def">
  <div class="cb-def-what">
    <span class="cb-def-label">What</span>
    <p>In Obsidian parlance, a folder of markdown files + a `.obsidian/` config directory that behaves as a self-contained knowledge base. (Obsidian is the first-class example; any folder of markdown qualifies.)</p>
  </div>
  <div class="cb-def-why">
    <span class="cb-def-label">Why it matters</span>
    <p>The vault is the <em>unit of content</em> cyberbaser operates on. It's the durable artifact; cyberbaser is the delivery mechanism. If cyberbaser disappears, the vault still opens and works.</p>
  </div>
  <div class="cb-def-how">
    <span class="cb-def-label">How cyberbaser uses it</span>
    <p>The cyberbase vault lives in its own repo (<code>cybersader/cyberbase</code>). Cyberbaser reads from it (a submodule or a build-time clone) and publishes it. The vault must stay a fully functional Obsidian vault, independent of cyberbaser.</p>
  </div>
</div>

## Single Source of Truth (SSOT)

<div class="cb-def">
  <div class="cb-def-what">
    <span class="cb-def-label">What</span>
    <p>The principle that exactly one authoritative place exists for any given piece of content. Every other representation is derived.</p>
  </div>
  <div class="cb-def-why">
    <span class="cb-def-label">Why it matters</span>
    <p>Prevents drift between representations, makes conflict resolution tractable (one side always wins), and keeps the mental model of the system simple.</p>
  </div>
  <div class="cb-def-how">
    <span class="cb-def-label">How cyberbaser uses it</span>
    <p><code>cybersader/cyberbase</code> is the SSOT. The web wiki is derived at build time. The Obsidian vault (local files on the maintainer's machine) is a read-view of the repo's current <code>main</code>. The Web CMS produces edits against the repo; it holds no state of its own.</p>
  </div>
</div>

## Wikilink

<div class="cb-def">
  <div class="cb-def-what">
    <span class="cb-def-label">What</span>
    <p>`[[Page Name]]` syntax that resolves to a link based on <em>page title</em>, not file path. Optional alias: `[[Page Name|display text]]`. Invented by wiki systems, canonicalized by Obsidian.</p>
  </div>
  <div class="cb-def-why">
    <span class="cb-def-label">Why it matters</span>
    <p>Standard markdown links (`[text](path/to/file.md)`) break whenever a file moves. Wikilinks survive reorganization because they reference identity, not location. For a growing knowledge base this is the difference between a living wiki and a graveyard of broken links.</p>
  </div>
  <div class="cb-def-how">
    <span class="cb-def-label">How cyberbaser uses it</span>
    <p>Wikilinks must round-trip cleanly. A Web CMS edit that produces `[[X]]` must be identical to an Obsidian edit that produces `[[X]]`. If the translation layer ever flattens to `[X](x.md)` the principle of round-trip fidelity is broken.</p>
  </div>
</div>

## Backlink

<div class="cb-def">
  <div class="cb-def-what">
    <span class="cb-def-label">What</span>
    <p>An automatically generated "pages that link to this one" list, built by looking at every wikilink across the whole vault.</p>
  </div>
  <div class="cb-def-why">
    <span class="cb-def-label">Why it matters</span>
    <p>Backlinks turn a pile of separate pages into a connected web. Readers find related pages without searching; authors see what points at what. This is the feature that makes a wiki feel like a wiki, not a flat docs site.</p>
  </div>
  <div class="cb-def-how">
    <span class="cb-def-label">How cyberbaser uses it</span>
    <p>Backlinks are generated automatically: scan every page for its links, then show, on each page, the list of pages that point to it (at the foot of the content or in a side panel). Some site generators do part of this already; making it work the same no matter which one you publish with is the translation layer's job, not any single renderer's.</p>
  </div>
</div>

## Translation Layer

<div class="cb-def cb-def-hard">
  <div class="cb-def-what">
    <span class="cb-def-label">What</span>
    <p>The step that turns Obsidian-flavored markdown into a web page, while keeping enough of its meaning intact that an edit made on the web can be saved back to the source without garbling it.</p>
  </div>
  <div class="cb-def-why">
    <span class="cb-def-label">Why it matters</span>
    <p>This is the single hardest and most load-bearing concept in the project. Get this right and the contribution paths compose cleanly; get it wrong and "round-trip editability" is an aspiration instead of a property.</p>
  </div>
  <div class="cb-def-how">
    <span class="cb-def-label">How cyberbaser uses it</span>
    <p>A tiered feature list: Tier 1 (full support, must round-trip), Tier 2 (partial, best-effort), Tier 3 (not supported, documented workaround). Each Tier 1 feature gets a fixture page + expected output + a round-trip correctness snapshot.</p>
  </div>
</div>

## Round-Trip Editability

<div class="cb-def cb-def-hard">
  <div class="cb-def-what">
    <span class="cb-def-label">What</span>
    <p>The property that a contributor can edit the same page in Obsidian, in the Web CMS, or on GitHub, and the other surfaces see the change without corruption.</p>
  </div>
  <div class="cb-def-why">
    <span class="cb-def-label">Why it matters</span>
    <p>Without this, contributors get locked into whichever surface they started with, and the whole "three independent paths" vision collapses. It's also the only way the maintainer can keep using Obsidian as their primary tool without dreading web contributions.</p>
  </div>
  <div class="cb-def-how">
    <span class="cb-def-label">How cyberbaser uses it</span>
    <p>Constrains every translation-layer decision. No lossy transforms. No "CMS-specific" schema that differs from the vault schema. Every Tier 1 feature gets a round-trip test fixture.</p>
  </div>
</div>

<p class="cb-fig-cap">Edit in any one of these. The other two read back the exact same file.</p>

## Open Authoring

<div class="cb-def">
  <div class="cb-def-what">
    <span class="cb-def-label">What</span>
    <p>A CMS pattern (popularized by Decap/Netlify CMS, continued by Sveltia) where an unauthenticated or lightly-authenticated user can propose changes via a forked-PR workflow, without needing direct commit access to the source repo.</p>
  </div>
  <div class="cb-def-why">
    <span class="cb-def-label">Why it matters</span>
    <p>Removes the biggest contribution barrier. Without Open Authoring, "contributors shouldn't need git" is impossible: you'd either need direct write access (a security problem) or contributors would have to fork manually (exactly the friction we're eliminating).</p>
  </div>
  <div class="cb-def-how">
    <span class="cb-def-label">How cyberbaser uses it</span>
    <p>Primary contribution path for the Web CMS entry point. Decap (or Sveltia) handles the fork-and-PR dance; a small OAuth proxy handles the auth handshake. The contributor never sees git.</p>
  </div>
</div>
