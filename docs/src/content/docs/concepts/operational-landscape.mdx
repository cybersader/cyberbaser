The operational landscape shows **who does what** in the contributable wiki ecosystem. It combines three dimensions:
- **Actors** — who builds, configures, contributes, reads, hosts
- **Components** — translation layer, CMS, auth, publishing, content, plugins
- **Resource intensity** — build-once investment vs periodic updates vs continuous operational work

This is the combined view that determines what to build into software, what to crowdsource, and what requires ongoing resources (money, people, tooling).

See [crosswalker's operational landscape](https://cybersader.github.io/crosswalker/concepts/operational-landscape/) for the sibling project's version of this analysis.

## The five tiers of work

### Tier 1: First-principles software <span style="color:#10b981">(build once)</span>

Work rooted in the fundamental nature of the problem. Once solved correctly, this rarely changes because the underlying principles don't change.

| Work item | Why it's resilient | Related |
|---|---|---|
| [Translation layer spec](/cyberbaser/design/translation-layer/) (tier system, round-trip rules) | Obsidian-flavored markdown is stable; the tier definitions are structural | [Principles #2](/cyberbaser/getting-started/principles/#2-obsidian-semantics-must-round-trip) |
| Remark/rehype plugins (wikilinks, callouts, embeds) | unified AST ecosystem is mature; plugin interfaces rarely break | [Ecosystem](/cyberbaser/concepts/ecosystem/) |
| [Round-trip correctness test suite](/cyberbaser/agent-context/zz-challenges/translation-layer-round-trip-enforcement/) | Equivalence rules for markdown are mathematical; fixture corpus is durable | [Translation layer](/cyberbaser/design/translation-layer/) |
| Content addressing / permalink scheme | URL stability is a wiki cardinal sin to violate; the scheme is chosen once | [Primitives: stable URLs](/cyberbaser/concepts/primitives/#content-addressable--stable-urls) |

**Business analogy**: R&D. High upfront cost, near-zero marginal cost afterward. This is the technical moat — if cyberbaser's translation layer is the best, it's reusable as a library (`@cyberbaser/translate`).

### Tier 2: Platform engineering <span style="color:#3b82f6">(build and refine)</span>

Built on first principles but adapts as the ecosystem shifts. Changes on the order of months to years.

| Work item | What causes change | Related |
|---|---|---|
| CMS integration ([Decap](https://decapcms.org/), [Sveltia](https://github.com/sveltia/sveltia-cms), [EmDash](https://github.com/emdash-cms/emdash)) | CMS landscape evolves; new features, new APIs | [Roadmap: CMS survey](/cyberbaser/getting-started/roadmap/) |
| Auth + OAuth proxy | Auth standards evolve; new providers emerge | [Roadmap: auth model](/cyberbaser/getting-started/roadmap/) |
| [Inline editing UX](/cyberbaser/getting-started/roadmap/) | Editor tech improves ([BlockSuite](https://blocksuite.io/), [Tiptap](https://tiptap.dev/)) | [Vault vision](/cyberbaser/agent-context/zz-log/2026-04-13-vault-vision-mining/) |
| Publishing pipeline (Astro + Starlight) | Astro/Starlight version updates, new integrations | [Architecture](/cyberbaser/design/architecture/) |
| Plugin abstraction ([Obsidian CLI/headless](https://obsidian.md/cli)) | Obsidian's headless runtime evolves; community plugins adapt | [Roadmap: plugin execution](/cyberbaser/getting-started/roadmap/) |

**Business analogy**: Product engineering. This is the **platform value** — the integration work that vault owners don't want to do themselves. A hosted service ("bring your vault, get a contributable wiki") is Tier 2 work packaged as a product.

### Tier 3: Community-maintained <span style="color:#eab308">(shared work)</span>

Per-vault or per-plugin work that's shareable. Once someone figures it out, everyone benefits.

| Work item | Who does it | Related |
|---|---|---|
| Plugin compatibility profiles (which Obsidian plugins work in headless mode) | Community tests; cyberbaser validates | [Roadmap: plugin execution](/cyberbaser/getting-started/roadmap/) |
| Vault templates / starter configs | First vault owner configures; shares via template repo | [Existing work](/cyberbaser/reference/existing-work/) |
| Contribution guidelines + PR templates | Community refines; vault owner customizes | [Contribution workflows](/cyberbaser/design/contribution-workflows/) |
| Theme variants (beyond silver+emerald) | Community contributes; cyberbaser maintains as examples | Current: [brand.css](/cyberbaser/development/) |

**Business analogy**: Marketplace / ecosystem. The more contributors, the more valuable each contribution. Like Obsidian's community plugin ecosystem — once someone builds a plugin, thousands of vaults benefit.

**Key insight**: This is where a community config registry (like [crosswalker's config browser](https://cybersader.github.io/crosswalker/features/config-system/)) would create the most value — per-vault configuration done once, shared infinitely.

### Tier 4: Operational maintenance <span style="color:#ef4444">(ongoing)</span>

Work that recurs because the external world changes. Cannot be eliminated — only automated or made more efficient.

| Work item | Why it recurs | Related |
|---|---|---|
| Hosting + deployment | Cloudflare Pages Functions, domain management, SSL | [Architecture](/cyberbaser/design/architecture/) |
| Dependency updates (Astro, Starlight, CMS, plugins) | Upstream releases on their own schedule; security patches | [Development](/cyberbaser/development/) |
| Content moderation (spam, abuse, PR review) | Abuse scales with openness; Open Authoring = open attack surface | [Contribution workflows](/cyberbaser/design/contribution-workflows/) |
| OAuth proxy maintenance | Token handling, secret rotation, provider API changes | [Roadmap: auth model](/cyberbaser/getting-started/roadmap/) |

**Business analogy**: DevOps / SRE. If cyberbaser were a business, this is where **subscription revenue** funds ongoing labor. A hosted service wraps Tier 4 into a monthly fee so vault owners don't have to manage it.

### Tier 5: User work <span style="color:#a855f7">(your effort)</span>

Work specific to YOUR vault. No tool can eliminate it — only make it more efficient.

| Work item | Why only you can do it | Related |
|---|---|---|
| Content creation | Only you know what to write about | The whole point |
| Contribution review | Only you can judge quality and accuracy for your domain | [Contribution workflows](/cyberbaser/design/contribution-workflows/) |
| Vault organization | Only you know the right structure for your knowledge domain | [Primitives: vault](/cyberbaser/concepts/primitives/#vault) |
| Contributor management | Only you decide who to trust with auto-merge vs manual review | [Contribution workflows](/cyberbaser/design/contribution-workflows/) |

**Business analogy**: Customer's work. Cyberbaser makes contribution FASTER and more reliable, but the actual knowledge is irreducibly human.

## The sustainability question

If cyberbaser were a business, where would the money come from?

| Tier | Funding model | What it looks like |
|---|---|---|
| **Tier 1** (first-principles software) | Open-source + grants + sponsorship | The translation layer is MIT-licensed. Anyone can use it. Goodwill + ecosystem gravity. |
| **Tier 2** (platform engineering) | **Hosted service (SaaS)** | "Bring your vault, get a contributable wiki." Monthly fee covers CMS setup, auth, inline editing, deployment. This is the product. |
| **Tier 3** (community) | Self-sustaining once critical mass is reached | Plugin compat profiles, vault templates, contribution guidelines — all community-contributed. |
| **Tier 4** (operations) | **Included in the SaaS fee** | Hosting, dependency updates, content moderation tooling, OAuth proxy — vault owners don't want to manage this. |
| **Tier 5** (user work) | **The value proposition** | Cyberbaser sells efficiency, not elimination. "Your wiki, your content, but anyone can contribute without learning git." |

**The core business thesis**: Tier 1 is the technical moat (hard to replicate). Tier 2+4 packaged together is the product (a hosted contributable wiki service). Tier 3 is the ecosystem (community contributions make the platform more valuable). Tier 5 is what the customer actually pays for — making their irreducible work as painless as possible.

**Comparison to existing models**:
- [Obsidian Publish](https://obsidian.md/publish) charges $10/month for Tier 2+4 — but has NO contribution model (no Tier 5 efficiency)
- [GitBook](https://www.gitbook.com/) charges per user for Tier 2+4 — but has NO Obsidian fidelity (no Tier 1 moat)
- [Notion public pages](https://www.notion.so/) is free for read — but has NO git SSOT (violates [Principle 1](/cyberbaser/getting-started/principles/#1-github-is-the-single-source-of-truth))
- Cyberbaser occupies the intersection: Obsidian fidelity + git SSOT + zero-git contribution + hosted service

## What this means for the roadmap

The [Research & Foundations roadmap](/cyberbaser/getting-started/roadmap/) should be read through this lens:

- **Identity question** → determines which tiers to invest in first
- **CMS survey** → Tier 2 decision (the platform choice)
- **Auth model** → Tier 2+4 decision (infrastructure + ongoing maintenance)
- **Plugin abstraction** → Tier 1 investment (if solved, becomes the technical moat)
- **Collaboration model** → Tier 2 scope (real-time collab = much more Tier 4 ops work)
- **Translation layer** → Tier 1 investment (the foundation everything else builds on)
- **Operational model** → THIS PAGE (which tiers get resourced, in what order)

**The architecture goal**: Push as much work as possible toward Tier 1 (solved once, resilient). Where that's impossible (Tiers 3-4), build community infrastructure so the work is shared. Only Tier 5 is irreducibly the user's burden.
