## TL;DR

- **Author from anywhere**: Obsidian, in-page web edits, a web CMS, GitHub. The **hub translates between every surface** into one faithful set of markdown files (round-trip, no corruption) and **governs** contributions (the maintainer's trust curve + a moderation queue). That translation between surfaces is the whole point.
- **The renderer is swappable**: turning the vault into a website (Quartz/Starlight) is a free commodity ([Quartz already wins it](/cyberbaser/research/assumptions-and-risks/#tested-quartz-prior-art-audit-2026-06-19)); never couple to one.
- **Hub-and-spoke**: one hub, many [vault-spokes](/cyberbaser/getting-started/vision/#the-shape-one-hub-many-spokes). **v1** = one hub, a few vaults; **v2+** = the same hub over a [federation](/cyberbaser/research/source-of-truth/#federation-mixed-privacy-and-role-based-access).

## How it all fits together

**The mental model:** write wherever you like → the hub keeps every surface the *same* faithful markdown and reviews each change → that markdown, with its full edit history, is the single **source of truth** → the published wiki is just a generated view of it.

**Under the hood:** your content is one folder of plain markdown files kept under **version control** (today that's *git*): every edit is saved for good, the full history is there, and you own the whole thing. The published wiki is just a website **generated** from those files (and the generator is swappable). Nothing here is locked to one app or one company.

## Boundaries: what each part must NOT do

The separations that keep the system honest:

| Part | Must **not** |
|---|---|
| **Vault (SSOT)** | hold build artifacts · depend on cyberbaser plugins to work in Obsidian · let its file layout be dictated by the URL scheme |
| **Hub / round-trip** | apply lossy transforms · be coupled to a specific renderer · hold content state outside the commit it produces |
| **Spoke renderer** | be treated as permanent or unique (it's swappable) · hold any contribution/auth state |
| **Edit surfaces** | require Obsidian to be running · keep content outside the vault · make the maintainer do anything special per edit |

## Open architectural questions

See [Open Questions](/cyberbaser/reference/open-questions/) for the full list; the architecturally-significant ones now:

- **Adopt a commodity renderer or keep the bespoke stack?** The [Quartz audit](/cyberbaser/research/assumptions-and-risks/#tested-quartz-prior-art-audit-2026-06-19) says the forward render is free and better elsewhere. Worth a spike before more Starlight investment.
- **Where does the hub runtime live?** It needs auth + write-back + moderation state; the [self-hosted Forgejo + PKCE stack](/cyberbaser/design/reference-architectures/self-hosted-forgejo-auth/) is one answer (and eliminates the OAuth proxy).
- **How does the round-trip resolve assets and block-IDs** when a web edit touches embeds (`![[note#^id]]`)? The one genuinely unsolved Tier-1 item.
- **What's the incremental build story** for a vault past ~1000 pages, and does a federation of vaults change it?
