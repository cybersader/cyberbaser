### Research tasks

#### R01 · What is cyberbaser's identity? <span class="cb-pill cb-pill-research">Open</span>

**Question:** Is cyberbaser a publishing tool, a wiki platform, a collaboration framework, or something else entirely? The vault research shows ambitions spanning all three — but the positioning needs to be ONE thing with clear boundaries.

**Why it matters:** Every downstream decision (CMS choice, auth model, plugin strategy) depends on what cyberbaser IS. Without this, we build features that don't compose.

#### R03 · Auth model <span class="cb-pill cb-pill-research">Open</span>

**Question:** GitHub OAuth only, or multi-provider auth? The vault has 20+ auth solutions researched. Need to decide scope for v1.

**Why it matters:** Auth gates the contribution path. GitHub-only is simple but excludes non-GitHub users. Multi-provider is flexible but adds complexity. The decision constrains the CMS choice (R02) and the OAuth proxy architecture.

**Candidates:** GitHub OAuth (current assumption), Better-Auth, Clerk, Ory/Kratos, SuperTokens, Auth.js, Auth0, Hanko, FusionAuth, Stytch, Stack Auth, OpenAuth, Keycloak, Authentik, Authelia, Bitwarden Passwordless, Fief, Cerbos, Oso, Dex

#### R04 · Plugin execution abstraction <span class="cb-pill cb-pill-research">Open</span>

**Question:** How can Obsidian community plugins (Dataview, linting, attachment management, etc.) run outside of Obsidian — in GitHub Actions, serverless functions, or a client-side CMS?

**Why it matters:** This is the "hardest layer" the user identified in the [vault vision doc](/cyberbaser/agent-context/zz-log/2026-04-13-vault-vision-mining/). Without this, contributors who don't use Obsidian get a degraded experience — plugin-managed patterns (tags, links, metadata) only work when someone opens the vault in Obsidian. That's a bottleneck.

**Major development (2025-2026):** Obsidian now has an **official headless runtime + CLI tool**:
- [Obsidian CLI](https://obsidian.md/cli) — ships with Obsidian 1.12 (Early Access, Catalyst license). "Anything you can do in Obsidian you can do from the command line."
- [Obsidian Headless](https://github.com/obsidianmd/obsidian-headless) — official headless client for Obsidian Sync and Publish. Sync/publish from CLI without the desktop app.
- [obsidianless](https://github.com/lucastraba/obsidianless) — community tool to run Obsidian CLI on headless Linux servers (no monitor, no subscription).
- [Headless Sync docs](https://help.obsidian.md/headless) · [CLI docs](https://obsidian.md/cli)

This potentially **solves the hardest problem.** If community plugins can run via the CLI in a GitHub Action, the "plugin execution across environments" gap closes. Key question: do community plugins actually load and execute in headless/CLI mode, or is it limited to core Obsidian features?

**Source material:** Vault: [Contributable Obsidian Wiki](/cyberbaser/agent-context/zz-log/2026-04-13-vault-vision-mining/) · [Obsidian headless GitHub issue #8](https://github.com/obsidianmd/obsidian-headless/issues/8) (headless server/Docker support) · [HN discussion](https://news.ycombinator.com/item?id=47197267)

**Decision log:** pending (needs hands-on testing of CLI + community plugins)

#### R05 · Collaboration model <span class="cb-pill cb-pill-research">Open</span>

**Question:** Git-only async collaboration, or real-time live editing? If real-time, which approach — CRDT (Yjs, Automerge), OT (ShareDB), or file-level sync (LiveSync, Peerdraft)?

**Why it matters:** The vault vision includes "edit with someone → allows you to enter an anonymous secure page to edit with someone live." That's a fundamentally different architecture from git-only PR workflows. Need to decide if v1 includes real-time collab or defers it.

**Candidates:**
- **Git-only** (simplest, current assumption): PRs, no realtime
- **CRDT-based**: Yjs, Automerge, SyncedStore, BlockSuite/OctoBase (from AFFiNE)
- **OT-based**: ShareDB, Convergence
- **File-level sync**: Peerdraft, LiveSync, SyncTrayzor
- **Hybrid**: ElectricSQL, PowerSync, RxDB, Turso

**Source material:** Vault: [Collaboration in Obsidian](/cyberbaser/agent-context/zz-log/2026-04-13-vault-collab-mining/) · [Cyberbase Development] sync/CRDT section (50+ links)

**Decision log:** [2026-04-13 · Vault collab mining](/cyberbaser/agent-context/zz-log/2026-04-13-vault-collab-mining/)

#### R06 · Translation layer deep-dive <span class="cb-pill cb-pill-research">Open</span>

**Question:** Does `astro-loader-obsidian` handle Tier 1 features (wikilinks, callouts, embeds, math, Mermaid) sufficiently, or do we need custom remark/rehype plugins?

**Why it matters:** The translation layer is the core technical differentiator. If the existing loader covers 80% of Tier 1, we're in "integration" territory. If it covers 30%, we're in "build from scratch" territory. Changes the Phase 1 scope by 5-10x.

**Source material:** Vault: [Astro with Content Layer] · Current `astro-loader-obsidian` dep in `docs/package.json` · [Obsidian Flavored Markdown spec](https://help.obsidian.md/Editing+and+formatting/Obsidian+Flavored+Markdown)

**Decision log:** pending (requires hands-on testing with real vault content)

#### R08 · Data sync strategy <span class="cb-pill cb-pill-research">Open</span>

**Question:** How does content flow between Obsidian (local files) → GitHub (repo) → Web CMS (edits) → Built site? What about media/images at scale?

**Why it matters:** Sync failures = lost contributions or corrupt content. The vault identified specific issues: Obsidian Git conflicts from date-modified fields, Windows path length limits, bracket escaping in filenames. These are real blockers.

**Source material:** Vault: [🐛 Known Base Bugs, Issues] · [Media Hosting in Obsidian] · [GitHub Repo Size Limits] · [Cyberbase Development] sync engines section

**Decision log:** pending

#### R09 · SEO for Obsidian-based sites <span class="cb-pill cb-pill-research">Open</span>

**Source material:** Vault: [Obsidian Publish SEO] · Starlight's built-in sitemap plugin · Current `@astrojs/sitemap` config
