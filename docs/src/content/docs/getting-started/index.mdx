## What cyberbaser is

Cyberbaser is **an interoperability layer for contributable, version-controlled knowledge bases.** A central **hub** sits between the surfaces people author from (Obsidian, an edit button on the page, a web CMS, GitHub), the renderers that publish the result, and other knowledge bases, keeping every change faithful as it round-trips between them. CMS-like contribution and a public wiki are things the layer enables, not the whole of what it is. It works on any markdown content, **not just one domain.** (We dogfood it on a cybersecurity vault, [`cybersader/cyberbase`](https://github.com/cybersader/cyberbase); that's just the first content, not the point.)

**The mental model:** [write wherever you like → the hub translates every surface into the *same* faithful markdown and reviews each change → that markdown, with full history, is the single source of truth → the wiki is a generated view of it](/cyberbaser/design/architecture/).

Two repos, easy to confuse:

| Repo | Role |
|---|---|
| [`cybersader/cyberbaser`](https://github.com/cybersader/cyberbaser) (this repo) | The tooling **and** the thinking: the hub, translation layer, CMS integration, deployment, and the first-principles research |
| [`cybersader/cyberbase`](https://github.com/cybersader/cyberbase) | The example content vault we dogfood on: plain markdown, one example, not the scope |

## Current phase: Research & Foundations

Cyberbaser is in **Phase R**: building first-principles understanding before shipping implementation. The [Roadmap](/cyberbaser/getting-started/roadmap/) is explicit: the Phase-0 publishing prototype is parked, and no later phase gets concrete steps until the problem, principles, and key decisions have real content.

This is deliberate. The old approach jumped to code before the principles justified the choices; the reinit inverted that — write the thinking, then write the code. A lot of that thinking is now done (see the research findings below).
