This page is the findings brief for the SEO + URL stability research task. The question: which SEO primitives does Astro/Starlight give us for free, which require custom work, and what permalink scheme keeps wiki links alive when the vault is reorganized?

**Bottom line up front:** Starlight handles more than most people expect (canonical URLs, sitemap, meta description, title tags, sitemap link injection) but stops short of OG images, JSON-LD, robots.txt, and RSS — all of which are one integration or one component-override away. The hardest problem is not the SEO plumbing: it is **URL stability across vault reorganization**, which requires a deliberate design decision today, not at launch.

<Aside type="caution" title="Scope reminder">
The Phase-0 Astro + Starlight prototype is **parked** — this research is to inform the *future* publish pipeline design, not to immediately extend the prototype. Implementation follows a principle; the principle is being established here.
</Aside>

## What Starlight gives you natively

When `site` is set in `astro.config.mjs`, Starlight (via its bundled `@astrojs/sitemap` dependency) provides these out of the box with zero additional configuration:

| Feature | Behavior |
|---|---|
| `<title>` tags | Page title + site title, joined by a configurable delimiter. Required. |
| `<meta name="description">` | Falls back to the global site `description` if page frontmatter omits it. |
| `<link rel="canonical">` | Auto-generated from the page URL. Frontmatter `slug` override deduplicated correctly (Starlight changelog confirms dedup logic was explicitly added). |
| Sitemap (`sitemap-index.xml`) | Generated at build time. Requires `site` URL. Linked via `<link rel="sitemap">` injected into every page head automatically. |
| `<link rel="sitemap">` head injection | Yes — Starlight injects this; frontmatter or global `head` override can suppress it. |
| Site-level `head` injection | Any tag can be added globally via the `head` array in `starlight({})` config — analytics scripts, verification tags, etc. |
| Per-page head override | Frontmatter `head` field accepts raw tag objects, including custom OG or canonical overrides. |
| Page/content hierarchy | Sidebar, breadcrumbs, next/prev links — visible navigation, not SEO-tagged (`rel="prev/next"` is NOT emitted). |

**What Starlight does NOT provide natively:**

- OG (`og:*`) meta tags — no default whatsoever
- Twitter/X card tags
- JSON-LD / structured data
- `robots.txt`
- RSS feed
- `BreadcrumbList` or any other schema markup
- `rel="prev"` / `rel="next"` on paginated content
- `llms.txt` (GEO / AI crawler hint file)
- Per-page OG image generation

## What requires custom work (and how hard each is)

### robots.txt — low effort

Astro has no native robots.txt generator. Two options:

1. **Static file** — drop `public/robots.txt` with `Sitemap: https://yoursite.com/sitemap-index.xml` and whatever crawl rules you need. Zero build complexity. Works on Cloudflare Pages with no adapter.
2. **Dynamic endpoint** — `src/pages/robots.txt.ts` with a `GET` export lets you generate it from config at build time. Useful if the site URL is an env variable.

Community package `astro-robots-txt` (npm: `astro-robots-txt`) wraps option 2 with more policy knobs. Not needed for v1; a static file is sufficient.

**v1 recommendation:** static `public/robots.txt`. Add `Sitemap:` directive pointing to `sitemap-index.xml`. Add AI-crawler `User-agent: GPTBot` and similar blocks or allowlists as desired (the 2026 checklist explicitly flags this).

### OG images — medium effort, two viable patterns

**Option A: astro-og-canvas (build-time, no external service)**

Creates a PNG per page during `astro build` using `canvaskit-wasm`. Requires an endpoint file, a route data middleware, and the npm package. Approximately 40-80 lines of config code. Increases build time on large sites but has no runtime dependency.

**Option B: Cloudinary Astro SDK (remote, no build-time cost)**

Constructs Cloudinary-hosted image URLs via the SDK. Zero build-time image generation; the URL is computed during page render. Requires a Cloudinary account and a cloud name in `.env`.

For a wiki with hundreds of pages, Option A (build-time) is safer long-term: no external service dependency, images work in offline/local preview, and links remain stable even if a cloud account changes.

**v1 recommendation:** defer per-page OG images to v1.1. For v1, set a single fallback `og:image` globally in the Starlight `head` config pointing to a static banner image in `public/`. This is three lines of config and takes 10 minutes. Dynamic per-page images can be added when the content volume justifies the build complexity.

### JSON-LD structured data — medium effort, stable pattern

Starlight's `Head` component can be overridden by registering a custom component in `astro.config.mjs`. The override receives `Astro.props.entry.data` (frontmatter) and `Astro.url.href`. The `schema-dts` npm package provides TypeScript types for schema.org types.

Recommended schema type for wiki content: `TechArticle` (for technical documentation pages) or `Article` for general content. Add `BreadcrumbList` as a second script tag using the page's sidebar hierarchy.

The override does not require a `<slot />` (the `Head` component builds an array internally, not a slot-based layout). The pattern calls `<Default {...Astro.props} />` then appends the `<script type="application/ld+json">` block.

**Stability caveat:** a Starlight maintainer hinted this API may get a first-class structured-data hook. Watch the changelog. The override pattern is not fragile — it has worked stably across multiple Starlight versions — but the props shape (`entry.data`) could change if Starlight revamps its route data model.

**v1 recommendation:** implement the Head override with a basic `TechArticle` schema. Add `BreadcrumbList` if breadcrumbs are visible in the UI (they are in Starlight's default theme). Validate with Google's Rich Results Test at launch.

### RSS feed — low-to-medium effort

`@astrojs/rss` is an official package. It requires:
- A `src/pages/rss.xml.js` (or `.ts`) endpoint
- Manual mapping of content collection entries to RSS item fields (`title`, `pubDate`, `description`, `link`)
- `site` URL configured in `astro.config.mjs`

For a documentation wiki rather than a blog, RSS is lower priority — users subscribe to "what changed" feeds, which requires `lastmod` metadata. The `@astrojs/sitemap` integration supports `lastmod` but the RSS feed needs frontmatter `updatedDate` fields to be useful.

**v1 recommendation:** defer RSS to v1.1. Add `updatedDate` to the content collection schema now so the data is ready when the feed is built.

### Canonical URLs — already handled

Starlight handles this natively. The only action needed is ensuring `site` is set in `astro.config.mjs`. Per-page `slug` overrides in frontmatter automatically flow through to the canonical tag (Starlight's dedup logic handles this). No custom work required.

### `rel="prev"` / `rel="next"` — not applicable to wiki

These are for sequential pagination (blog archives, search result pages). A wiki with a sidebar hierarchy does not need them. Skip.

## The URL stability problem: the real decision

**This is the most consequential SEO design decision for a wiki.** SEO is easy to bolt on later; broken inbound links are permanent damage.

The threat: a contributor moves `security/authentication/oauth.md` to `identity/protocols/oauth.md` because the vault's folder structure evolved. The URL changes from `/security/authentication/oauth` to `/identity/protocols/oauth`. Every inbound link — from other wikis, from Google's index, from blog posts — now 404s.

### Three options

#### Option 1: Slug-derived URLs (default Astro behavior)

URL is derived from file path: `docs/src/content/docs/security/authentication/oauth.md` → `/security/authentication/oauth`.

- Pro: human-readable, SEO-friendly, zero config
- Pro: file name = URL = easy to reason about
- Con: **any file move or rename breaks the URL**
- Con: requires a redirect entry for every reorganization, forever
- Con: redirect debt accumulates; large wikis end up with hundreds of stale redirects

This is WordPress's default behavior and is why large WordPress sites accumulate redirect chains.

#### Option 2: Frontmatter `permalink` (stable identifier in the file)

Each file carries a `permalink` field in frontmatter that becomes the canonical URL, regardless of where the file lives:

```markdown
---
title: OAuth 2.0
permalink: /concepts/oauth
---
```

- Pro: file can move anywhere; URL never changes
- Pro: matches the pattern Obsidian Publish adopted (their `permalink` frontmatter field)
- Pro: contributors can set URLs that match conceptual hierarchy, not folder structure
- Con: requires discipline to set on every page; new pages without `permalink` need a fallback
- Con: `permalink` values can collide if two contributors pick the same path
- Con: Astro does not natively use a `permalink` field — it would require a custom loader or `getStaticPaths` logic that reads frontmatter and maps to routes

**This is the approach Obsidian Publish uses natively** (the `permalink` key in Obsidian frontmatter is recognized by Obsidian Publish and controls the published URL). For a vault-to-web pipeline, reading this field in the Astro loader is a natural fit.

#### Option 3: UUID / content-hash stable IDs

Each file gets an opaque identifier (UUID or short hash) as its permalink:

```markdown
---
id: x2HYMKwttrCCoaznbBbSqb
---
```

URL becomes `/x2HYMKwttrCCoaznbBbSqb`. Human-unreadable.

This is the pattern described in [Stable URLs on Obsidian Publish](https://joschua.io/posts/2023/10/07/stable-publish-urls/) — the author used it specifically to eliminate redirect maintenance overhead.

- Pro: guaranteed uniqueness, zero collision risk
- Pro: zero redirect debt forever
- Con: terrible UX — URLs in shared links are opaque strings
- Con: loses all SEO keyword signal in the URL path
- Con: makes it hard to audit what URL a file produces without opening it

**Verdict: reject for a public wiki.** A wiki lives or dies on shareability. `/identity/protocols/oauth` tells a user exactly what they're getting; `/x2HYMKwttrCCoaznbBbSqb` does not.

### Recommended approach: frontmatter permalink + redirect table as escape hatch

**Primary mechanism:** Add a `permalink` field to the Starlight content collection schema. The Astro `getStaticPaths()` implementation reads `permalink` (if set) and falls back to the file path slug. This gives contributors explicit URL control without requiring it.

**Escape hatch for reorganizations:** Maintain a `_redirects` file (for Cloudflare Pages) as the source of truth for all retired URLs. Cloudflare Pages supports up to 2,100 redirects in `_redirects` (2,000 static + 100 dynamic). For larger wikis, Cloudflare Bulk Redirects (available at the account level via the Cloudflare dashboard) handles hundreds of thousands of rules with no per-rule limit.

**Critical caveat on Astro's `redirects` config in static/SSG mode:** When building to static HTML without an adapter (Cloudflare Pages with static assets — no Worker), Astro's `redirects:` config in `astro.config.mjs` produces `<meta http-equiv="refresh">` HTML pages that return HTTP 200, not 301. This is an SEO antipattern — Google treats it differently from a real 301, and the status code is lost. The correct approach on Cloudflare Pages static deployments is to write redirect rules directly to `public/_redirects`. A community integration script (by Alex Duggleby) hooks into `astro:build:done` to auto-convert Astro `redirects:` config into Cloudflare `_redirects` format, allowing centralized management.

**Redirect precedence on Cloudflare Pages:** static rules (exact match) before dynamic (splat/placeholder). Redirects are applied before asset serving — an asset at `/old-path/index.html` will still be redirected if a rule matches.

### The `redirect_from` / `aliases` pattern

A wiki-friendly pattern from Jekyll and Hugo: pages declare their previous URLs in frontmatter:

```markdown
---
title: OAuth 2.0
permalink: /concepts/oauth
redirect_from:
  - /security/authentication/oauth
  - /auth/oauth
---
```

A build-time step reads these fields and emits redirect rules. This keeps the redirect logic with the content file itself rather than in a separate config file, making it much easier for contributors to self-serve URL migrations.

`astro-redirect-from` (community package) implements this pattern for Astro. It generates Astro config redirects from frontmatter — though see the caveat above about static mode outputting `<meta>` redirects instead of 301s on Cloudflare Pages. A custom integration that emits `_redirects` entries instead of config redirects would be needed for proper HTTP 301 behavior.

Items are ranked by impact and whether they are pre-launch blockers vs. post-launch improvements.

### Pre-launch (must-have for v1)

| # | Item | Native? | Action |
|---|---|---|---|
| 1 | `site` URL set in `astro.config.mjs` | — | Config. Unlocks sitemap + canonicals. |
| 2 | Canonical URLs | Yes (Starlight) | Set `site`. Done. |
| 3 | Title + meta description on every page | Yes (Starlight) | Require `description` in content schema (Zod). |
| 4 | Sitemap (`sitemap-index.xml`) | Yes (Starlight) | Set `site`. Confirm link injection in head. |
| 5 | `robots.txt` | No | Static `public/robots.txt`. Add Sitemap directive. |
| 6 | Global fallback `og:image` | No | Add to Starlight `head` config. Static image, 10 min. |
| 7 | `og:title` + `og:description` + `og:url` | No | Head component override or per-page frontmatter `head`. |
| 8 | Basic JSON-LD (`TechArticle`) | No | Head component override. ~50 lines. |
| 9 | 301 redirects for any retired URLs | No | `public/_redirects`. HTTP 301 via Cloudflare. |
| 10 | Permalink scheme decided | No | Adopt frontmatter `permalink` field. Add to schema. |
| 11 | `redirect_from` field in schema | No | Add to content schema. Wire up build step. |

### Post-launch (v1.1 improvements)

| # | Item | Notes |
|---|---|---|
| 12 | Per-page OG images | `astro-og-canvas` at build time. Increases build time. |
| 13 | RSS feed | Requires `updatedDate` frontmatter field (add now). |
| 14 | `BreadcrumbList` JSON-LD | Extend Head override. |
| 15 | `rel="prev"` / `rel="next"` | Only if paginated content exists. |
| 16 | `llms.txt` | Static file. Low effort. Add when content is stable. |
| 17 | `noindex` on 404 | Add to 404 layout. |
| 18 | AI crawler bot rules | Extend `robots.txt` with explicit `User-agent:` blocks or allowlists. |
| 19 | `dateModified` visible on pages | Extend layout to show `updatedDate` when set. |

## Permalink scheme recommendation

**Adopt frontmatter `permalink` as the canonical URL contract.** Reasoning:

1. It matches what Obsidian Publish already uses — contributors familiar with Obsidian Publish will recognize the pattern, and vault files already using `permalink` will work without modification.
2. It decouples URL from folder structure — vault can be reorganized freely; only the permalink field is load-bearing for the published URL.
3. It keeps URLs human-readable and SEO-friendly (keyword-bearing, kebab-case paths).
4. It is implementable in Astro today via `getStaticPaths()` reading frontmatter.
5. Files without a `permalink` fall back to the file-path slug, so the system degrades gracefully for pages that have never been reorganized.

The `redirect_from` companion field handles the migration path when a `permalink` is itself changed (rare but must be possible).

**Do not use content-hash / UUID URLs.** The UX cost to a public wiki outweighs the redirect maintenance savings.

**Do not rely on Astro `redirects:` config alone in static mode.** It emits `<meta http-equiv="refresh">` (HTTP 200), not HTTP 301. Use `public/_redirects` for Cloudflare Pages static deployments.

## Open questions

1. **Does `astro-loader-obsidian` read Obsidian's `permalink` frontmatter field?** If yes, the URL scheme is already compatible with vault files. Needs hands-on test.
2. **Build time impact of `astro-og-canvas`** on a 500+ page wiki. Needs benchmarking before committing to it for v1.
3. **Structured data API in Starlight** — the maintainers hinted at a first-class hook. Track the Starlight changelog before implementing a Head override.
4. **`redirect_from` field**: should it be a standard field in all cyberbaser-exported pages, or only added when a page is first renamed? Decision affects the content schema.
5. **Cloudflare Bulk Redirects** vs `_redirects` threshold: at what wiki size (number of pages reorganized in a major restructure) does Bulk Redirects become necessary? Estimate: a vault reorganization that touches 20% of 500 pages = 100 redirects — well within the 2,100 `_redirects` limit. Bulk Redirects is an escape hatch, not a v1 requirement.
